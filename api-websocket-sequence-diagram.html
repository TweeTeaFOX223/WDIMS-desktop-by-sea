WDIMS の API / WebSocket 動作分析

このページは、実装コードを読み取って REST API と Socket.IO の流れを整理したものです。 中心は server/routes.ts の API 群、server/websocket.ts の同期イベント、そして最終的な保存先である config/ 配下の JSON / テキストファイルです。

HTTP サーバー: Hono + @hono/node-server リアルタイム同期: Socket.IO 保存先: config/profiles / config/browsers.json / active-profile.txt

REST の役割

初期表示、プロファイル管理、ブラウザ設定、明示的な保存処理を担当します。

WebSocket の役割

同じプロファイルを開いている別ウィンドウへ、設定変更を即時反映します。

重要な実態

設定の正本は REST API で、WebSocket は他クライアントへの同期通知だけを担当します。

1. 起動直後の読み込みフロー

`App.tsx` 初回マウント時の流れです。まずウィンドウ単位の localStorage を確認し、無ければサーバーのアクティブプロファイルを取得します。 

sequenceDiagram
    autonumber
    actor User as User
    participant App as client/App.tsx
    participant LS as localStorage
    participant API as REST /api
    participant PM as profileManager
    participant Files as config files
    participant WS as Socket.IO server

    User->>App: アプリを開く
    App->>LS: loadActiveProfile()
    alt localStorage にプロファイルあり
        LS-->>App: profileName
    else localStorage にプロファイルなし
        App->>API: GET /api/profiles/active/current
        API->>PM: getActiveProfile()
        PM->>Files: read config/active-profile.txt
        Files-->>PM: active profile
        PM-->>API: profileName
        API-->>App: { profileName }
        App->>LS: saveActiveProfile(profileName)
    end
    App->>API: GET /api/profiles/:name
    API->>PM: getProfile(profileName)
    PM->>Files: read *.wdims_ui.json
    PM->>Files: read *.wdims_engine.json
    Files-->>PM: displaySettings + searchEngines
    PM-->>API: Profile JSON
    API-->>App: profile data
    App->>WS: initializeWebSocket()
    WS-->>App: connect

2. 表示設定 / 検索エンジン設定の更新フロー

設定画面や編集操作の保存は、まず REST で保存し、その直後に WebSocket で他クライアントへ通知します。 現在の WebSocket サーバーは保存処理を持たず、通知専用です。 

sequenceDiagram
    autonumber
    actor User as 編集したユーザー
    participant AppA as client/App.tsx (送信元)
    participant API as REST /api
    participant PM as profileManager
    participant Files as profile JSON files
    participant WS as Socket.IO server
    participant AppB as 別クライアント

    User->>AppA: 設定変更 / タブ編集 / エンジン編集
    AppA->>AppA: setState() で即時反映
    alt 表示設定変更
        AppA->>API: PUT /api/profiles/:name/display-settings
        API->>PM: updateDisplaySettings(profileName, settings)
        PM->>Files: overwrite *.wdims_ui.json
        Files-->>PM: saved
        PM-->>API: ok
        API-->>AppA: 200 OK
        AppA->>WS: emit "display-settings-update"
        WS-->>AppA: emit "display-settings-update-success"
        WS-->>AppB: broadcast "display-settings-changed"
        AppB->>AppB: activeProfileName が一致すれば setSettings()
    else 検索エンジン/タブ変更
        AppA->>API: PUT /api/profiles/:name/search-engines
        API->>PM: updateSearchEngines(profileName, engines)
        PM->>Files: overwrite *.wdims_engine.json
        Files-->>PM: saved
        PM-->>API: ok
        API-->>AppA: 200 OK
        AppA->>WS: emit "search-engines-update"
        WS-->>AppA: emit "search-engines-update-success"
        WS-->>AppB: broadcast "search-engines-changed"
        AppB->>AppB: activeProfileName が一致すれば setConfig()
    end
この構成では、保存は REST のバリデーション付き経路に一本化されます。別クライアントは WebSocket で受けた内容を表示へ反映します。

3. プロファイル管理 API フロー

プロファイル作成、削除、複製、リネーム、切り替えはすべて REST 中心です。クライアント側の `profiles-changed` 通知は現状使われていません。 

sequenceDiagram
    autonumber
    actor User as User
    participant PMgr as client/ProfileManager.tsx
    participant API as REST /api
    participant PM as profileManager
    participant Files as config directory
    participant App as client/App.tsx

    alt 新規作成
        User->>PMgr: プロファイル作成
        PMgr->>API: POST /api/profiles
        API->>PM: createProfile(name, copyFrom?)
        PM->>Files: mkdir config/profiles/:name
        PM->>Files: write :name.wdims_ui.json / :name.wdims_engine.json
        API-->>PMgr: 201 Created
        PMgr->>API: GET /api/profiles
    else 削除
        User->>PMgr: プロファイル削除
        PMgr->>API: DELETE /api/profiles/:name
        API->>PM: deleteProfile(name)
        PM->>Files: rm profile directory
        API-->>PMgr: 200 OK
        PMgr->>API: GET /api/profiles
    else 複製
        User->>PMgr: プロファイル複製
        PMgr->>API: POST /api/profiles/:name/clone
        API->>PM: cloneProfile(source, newName)
        PM->>Files: read source files
        PM->>Files: write newName files
        API-->>PMgr: 201 Created
    else 名前変更
        User->>PMgr: プロファイル名変更
        PMgr->>API: PUT /api/profiles/:name/rename
        API->>PM: renameProfile(oldName, newName)
        PM->>Files: rename directory
        API-->>PMgr: 200 OK
        opt 変更対象が現在使用中
            PMgr->>API: PUT /api/profiles/active/current
            API->>PM: setActiveProfile(newName)
            PM->>Files: write config/active-profile.txt
            API-->>PMgr: 200 OK
            PMgr->>App: onProfileSwitch(newName)
            App->>API: GET /api/profiles/:name
        end
    else 切り替え
        User->>PMgr: 別プロファイルへ切り替え
        PMgr->>API: PUT /api/profiles/active/current
        API->>PM: setActiveProfile(profileName)
        PM->>Files: write config/active-profile.txt
        API-->>PMgr: 200 OK
        PMgr->>App: onProfileSwitch(profileName)
        App->>API: GET /api/profiles/:name
    end

4. ブラウザ設定 API とブラウザ起動

ブラウザ設定は WebSocket を使わず、`config/browsers.json` への読み書きと OS コマンド実行で完結します。 

sequenceDiagram
    autonumber
    actor User as User
    participant UI as client/BrowserSettings.tsx
    participant API as REST /api
    participant BL as browserLauncher
    participant Files as config/browsers.json
    participant OS as OS / child_process.exec

    User->>UI: ブラウザ設定画面を開く
    UI->>API: GET /api/browsers
    API->>BL: loadBrowsersConfig()
    BL->>Files: read config/browsers.json
    Files-->>BL: config
    BL-->>API: config
    API-->>UI: config JSON

    alt 設定保存
        User->>UI: 保存
        UI->>API: PUT /api/browsers
        API->>BL: saveBrowsersConfig(config)
        BL->>Files: overwrite config/browsers.json
        API-->>UI: 200 OK
    else 特定ブラウザ起動
        User->>UI: このブラウザでアプリを開く
        UI->>API: POST /api/browsers/launch/:browserName
        API->>BL: launchSpecificBrowser(browserName, defaultPort)
        BL->>Files: read config/browsers.json
        BL->>OS: exec(browserPath args http://localhost:port)
        OS-->>BL: success / failure
        BL-->>API: result
        API-->>UI: 200 OK or 400
    end

5. 実装済みだが現状ほぼ使われていない WebSocket イベント

  • profile-switch / profile-switched: サーバーとクライアントサービスには実装がありますが、現行 UI の主経路では使っていません。プロファイル切り替えは REST で行っています。
  • profiles-changed / profiles-list-changed: サーバーイベントはありますが、`ProfileManager.tsx` から emit されておらず、一覧更新は毎回 REST 再取得です。
  • *-update-success: 送信元に成功通知していますが、クライアント側で受信ハンドラを登録していません。
設計意図としては「複数ウィンドウ同期」がありますが、実際に UI が利用しているのは display-settings-changed と search-engines-changed の 2 系統が中心です。
